# Question Generation - 질문지 작성 방법론

## 목적
PRD에서 식별된 불명확한 영역에 대해 구조화되고 효과적인 질문을 생성하여, 사용자로부터 명확한 답변을 얻습니다.

## 사용 시기
- 불명확 영역에 대한 질문을 작성할 때
- 질문 형식이나 우선순위 결정에 도움이 필요할 때
- 참조 관련 질문을 작성할 때 (모듈화 프로젝트)
- 사용자가 이해하기 쉬운 질문 형태로 전환해야 할 때

## 핵심 원칙

### 1. 질문의 개수 제한
- **최대 5-7개 질문** per 라운드
- 우선순위가 높은 것부터 선택

### 2. 우선순위 기반 질문 선택

**Critical (즉시 해결 필요)**:
- 프로젝트의 핵심 목적
- 주요 기능의 입출력
- 아키텍처 결정
- 데이터 구조 (공통 사용)

**High (우선 해결)**:
- 세부 기능 로직
- 에러 처리 방식
- UI/UX 주요 플로우
- 기능 간 의존성

**Medium (중요)**:
- 세부 에러 케이스
- 부가 기능
- 최적화 방안
- 설정 옵션

**Low (선택적)**:
- 향후 확장성
- 마이너 UI 개선
- 로깅 상세
- 문서화 수준

### 3. 질문 형식 우선순위

1. **선택형 질문** (최우선)
2. **예시 요청형 질문**
3. **개방형 질문** (최소화)

## 질문 형식 가이드

### 형식 1: 선택형 질문 (권장)

구체적인 옵션을 제공하여 사용자가 쉽게 선택할 수 있게 합니다.

```markdown
### [문서명 | 영역] 질문 N: [질문 제목] [우선순위 태그]

**대상 문서**: `[문서명.md]` (모듈화 프로젝트만)  
**섹션**: [해당 섹션명] (모듈화 프로젝트만)

**상황**: 
[현재 PRD의 내용과 무엇이 불명확한지 구체적으로 설명]

**질문**: 
[구체적인 질문]

**옵션**:

A. [옵션 1 설명]
   - 장점: ...
   - 단점: ...
   - 예시: ...
   
B. [옵션 2 설명]
   - 장점: ...
   - 단점: ...
   - 예시: ...

C. [옵션 3 설명] (선택적)
   - 장점: ...
   - 단점: ...
   - 예시: ...

D. 기타 (구체적으로 설명해주세요)

**이 결정이 영향을 미치는 부분**:
- [영향 받는 기능/문서 1]: [구체적 영향]
- [영향 받는 기능/문서 2]: [구체적 영향]

**참조 필요 여부**: (모듈화 프로젝트만)
- [ ] 이 내용은 PRD-Shared.md에 정의되어야 함
- [ ] 이 내용은 현재 문서에만 해당됨
```

**좋은 예시**:
```markdown
### [핵심 기능] 질문 1: 입력 파일 형식 [Critical]

**상황**: 
"여러 형식의 파일을 지원한다"고 되어 있으나, 구체적인 형식이 명시되지 않았습니다.

**질문**: 
Phase 1에서 어떤 파일 형식을 지원해야 할까요?

**옵션**:

A. CSV만 지원 (최소)
   - 장점: 개발 빠름, 테스트 단순
   - 단점: 활용도 제한적
   - 예시: input.csv → 처리 → output.csv
   
B. CSV + JSON 지원
   - 장점: 웹 API 연동 가능, 구조화된 데이터
   - 단점: 파서 2개 필요
   - 예시: input.csv 또는 input.json → 처리
   
C. CSV + JSON + Excel 지원
   - 장점: 사용자 편의성 최대
   - 단점: 개발 시간 증가, Excel 라이브러리 의존성
   - 예시: .csv, .json, .xlsx 모두 처리

D. 기타 형식 (구체적으로 알려주세요)

**이 결정이 영향을 미치는 부분**:
- 파일 파서 모듈: 필요한 라이브러리 개수
- 테스트 케이스: 형식별 테스트 필요
- 사용자 가이드: 지원 형식 명시
- 개발 일정: 형식 추가마다 +2-3일
```

### 형식 2: 예시 요청형 질문

사용자가 구체적인 예시를 제공하도록 유도합니다.

```markdown
### [영역] 질문 N: [질문 제목] [우선순위 태그]

**상황**: 
[현재 PRD의 내용과 무엇이 불명확한지 설명]

**질문**: 
구체적인 예시를 제공해주실 수 있나요?

**원하는 예시 형식**:

**입력 예시**:
```
[입력 데이터 형식]
```

**기대 출력**:
```
[출력 데이터 형식]
```

**특수 케이스** (있다면):
```
[엣지 케이스 예시]
```

**이 예시가 명확히 할 부분**:
[예시를 통해 어떤 모호성이 해소될지 설명]
```

**좋은 예시**:
```markdown
### [데이터 구조] 질문 2: 번역 단위 구조 [Critical]

**상황**: 
"번역 단위"라는 용어가 여러 번 등장하지만, 정확히 어떤 데이터를 포함하는지 불명확합니다.

**질문**: 
실제 사용 예시를 들어주실 수 있나요?

**원하는 예시 형식**:

**입력 예시** (게임 텍스트):
```
원본 언어: 한국어
텍스트: "마을로 돌아가시겠습니까?"
컨텍스트: (어디에 사용? NPC 대화? UI 버튼?)
기타 정보: (캐릭터명? 변수? 태그?)
```

**기대 출력** (번역 후):
```
타겟 언어: 영어
번역: "Would you like to return to the village?"
번역 방식: (사람? 기계? TM?)
품질 점수: (있다면)
```

**특수 케이스** (있다면):
- 변수가 포함된 경우: "당신은 {level}레벨입니다"
- 복수형 처리가 필요한 경우
- 문화적 맥락이 중요한 경우

**이 예시가 명확히 할 부분**:
- 번역 단위에 포함될 필드 결정
- 메타데이터 구조 정의
- 특수 케이스 처리 방법
```

### 형식 3: 개방형 질문 (최소화)

선택지를 제공하기 어려운 경우에만 사용합니다.

```markdown
### [영역] 질문 N: [질문 제목] [우선순위 태그]

**상황**: 
[현재 PRD의 내용과 무엇이 불명확한지 설명]

**질문**: 
[구체적인 질문]

**답변 가이드**:
다음 측면을 고려하여 답변해주세요:
- [고려사항 1]
- [고려사항 2]
- [고려사항 3]

**참고사항**:
- [도움이 될 정보]
- [비슷한 사례]

**이 정보가 필요한 이유**:
[왜 이 정보가 PRD 완성에 중요한지 설명]
```

**좋은 예시**:
```markdown
### [에러 처리] 질문 3: 에러 처리 정책 [High]

**상황**: 
에러 처리 방식이 명시되지 않았습니다.

**질문**: 
다음 상황에서 프로그램이 어떻게 동작해야 할까요?

1. 입력 파일이 존재하지 않을 때
2. 파일 형식이 잘못되었을 때
3. 네트워크 오류로 API 호출 실패 시
4. 예상치 못한 데이터 형식일 때

**답변 가이드**:
각 상황에 대해:
- 에러 메시지를 보여주고 종료?
- 에러를 로그하고 계속 진행?
- 사용자에게 재시도 옵션 제공?
- 기본값으로 대체?

**참고사항**:
- 사용자가 기술자인지 일반 사용자인지에 따라 달라질 수 있음
- 배치 처리의 경우 한 건 실패가 전체에 영향을 주는지 고려

**이 정보가 필요한 이유**:
에러 처리 방식은 사용자 경험과 시스템 안정성에 직접적인 영향을 미칩니다.
```

### 형식 4: 문서 구조 관련 질문 (모듈화 프로젝트만)

기능이 어느 문서에 속해야 할지, 공통 요소를 어떻게 관리할지 질문합니다.

```markdown
### [문서 구조] 질문 N: [질문 제목] [우선순위]

**상황**: 
[기능/데이터가 어디에 속해야 할지 불명확함]

**질문**: 
이 [기능/데이터/정책]을 어떻게 문서화해야 할까요?

**옵션**:

A. 독립된 Feature 문서로 분리
   - 조건: [복잡도, 독립성 등]
   - 파일명: PRD-Feature-[Name].md
   - 장단점: ...
   
B. 기존 Feature 문서에 서브 섹션으로 포함
   - 대상 문서: PRD-Feature-[ExistingName].md
   - 이유: [기능 간 결합도 등]
   - 장단점: ...
   
C. Shared 문서에 공통 요소로 정의
   - 이유: 여러 기능에서 사용됨
   - 참조 방식: [링크 예시]
   - 장단점: ...

**영향**:
- 문서 개수 및 구조
- 기능 간 참조 관계
- 향후 유지보수
```

---

### 형식 5: 참조 구조 질문 (모듈화 프로젝트만)

중복 가능성이 있는 내용을 어떻게 관리할지 질문합니다.

```markdown
### [참조 구조] 질문 N: [데이터/정책명] 정의 위치 [우선순위]

**상황**: 
[데이터 구조/정책/용어]가 여러 기능에서 사용됩니다.

**현재 상태**:
- PRD-Feature-A.md에 정의되어 있음
- PRD-Feature-B.md에서도 사용됨
- PRD-Feature-C.md에서도 필요함

**질문**: 
이 [요소]를 어떻게 관리해야 할까요?

**권장 옵션**:

A. PRD-Shared.md로 이동 (강력 권장)
   - PRD-Shared.md의 "[적절한 섹션]"에 정의
   - 다른 문서에서는 참조 링크만 사용
   - 참조 예시:
     ```markdown
     [요소명]은 [PRD-Shared.md의 해당 섹션](PRD-Shared.md#섹션-id)을 참조하세요.
     ```
   - 장점: 단일 진실 공급원, 관리 편의
   - 단점: 문서 간 이동 필요
   
B. 원본 문서에 유지하고 다른 문서에서 참조
   - PRD-Feature-A.md에 정의 유지
   - PRD-Feature-B.md, C.md에서 A를 참조
   - 참조 예시:
     ```markdown
     [요소명]은 [PRD-Feature-A.md의 해당 섹션](PRD-Feature-A.md#섹션-id)을 참조하세요.
     ```
   - 장점: 기능과 함께 관리
   - 단점: 기능 간 결합도 증가

**영향**:
- 관리 복잡도
- 문서 간 결합도
- 향후 수정 시 영향 범위

**권장**: 
3개 이상의 문서에서 사용된다면 반드시 Shared로 이동
```

## 질문 작성 프로세스

### Step 1: 불명확 영역 우선순위 정렬

메인 스킬의 Phase 1에서 식별된 불명확 영역을 우선순위대로 정렬합니다.

```markdown
## 불명확 영역 우선순위

### Critical (5개)
1. [영역 1]
2. [영역 2]
3. [영역 3]
4. [영역 4]
5. [영역 5]

### High (3개)
1. [영역 6]
2. [영역 7]
3. [영역 8]

### Medium (2개)
...

### Low (1개)
...
```

---

### Step 2: 질문 개수 결정

최대 5-7개를 선택합니다.

**선택 전략**:
1. Critical 모두 포함 (5개)
2. 남은 슬롯에 High에서 선택 (2개)
3. 총 7개 질문

**라운드 계획**:
```markdown
## 질문 라운드 계획

### 라운드 1 (7개 질문)
- Critical: 5개
- High: 2개

### 라운드 2 (예상, 6개 질문)
- High: 1개 (남은 것)
- Medium: 2개
- Low: 1개
- 라운드 1 후속 질문: 2개

### 라운드 3 (예상, 필요시)
- 나머지 Medium/Low
- 이전 라운드 후속 질문
```

---

### Step 3: 각 질문의 형식 결정

각 불명확 영역에 가장 적합한 질문 형식을 선택합니다.

**결정 기준**:

| 불명확 영역 유형 | 권장 형식 |
|------------------|----------|
| 선택 가능한 옵션이 명확함 | 선택형 |
| 구체적 사례가 필요함 | 예시 요청형 |
| 매우 개방적이고 창의적 답변 필요 | 개방형 |
| 문서 배치 관련 | 문서 구조 질문 |
| 중복 요소 관리 | 참조 구조 질문 |

---

### Step 4: 질문 초안 작성

선택된 형식에 따라 질문을 작성합니다.

**체크리스트**:
- [ ] 상황 설명이 구체적인가?
- [ ] 질문이 명확한가?
- [ ] 옵션이 충분히 상세한가? (선택형)
- [ ] 장단점이 명시되었는가? (선택형)
- [ ] 예시 형식이 제공되었는가? (예시 요청형)
- [ ] 답변 가이드가 명확한가? (개방형)
- [ ] 영향 범위가 설명되었는가?
- [ ] 우선순위 태그가 있는가?

---

### Step 5: 질문 간 의존성 확인

일부 질문의 답변이 다른 질문에 영향을 줄 수 있습니다.

**의존성 예시**:
```markdown
질문 1: 아키텍처 선택 (모놀리식 vs 마이크로서비스)
  ↓ 영향
질문 3: API 설계 방식
  ↓ 영향
질문 5: 인증 메커니즘

→ 질문 1의 답변에 따라 질문 3, 5의 옵션이 달라짐
```

**대응 방법**:
```markdown
### 질문 1: 아키텍처 선택 [Critical]
...

### 질문 3: API 설계 [High]

**참고**: 이 질문은 질문 1의 답변에 따라 달라집니다.
- 질문 1에서 A 선택 시: [옵션 세트 1]
- 질문 1에서 B 선택 시: [옵션 세트 2]

먼저 질문 1에 답변해주시면, 질문 3을 구체화하겠습니다.
```

---

### Step 6: 질문 그룹화 (모듈화 프로젝트)

문서별로 질문을 그룹화하여 사용자가 컨텍스트를 유지하기 쉽게 합니다.

```markdown
## 질문 라운드 1

### Master 문서 관련 (2개)
#### 질문 1: [제목] [Critical]
...

#### 질문 2: [제목] [Critical]
...

### Shared 문서 관련 (2개)
#### 질문 3: [제목] [Critical]
...

#### 질문 4: [제목] [High]
...

### Feature 문서 관련

#### [Feature A] 질문 5: [제목] [Critical]
...

#### [Feature B] 질문 6: [제목] [High]
...

#### [Feature A] 질문 7: [제목] [High]
...
```

## 질문 작성 체크리스트

각 질문을 작성한 후 다음을 확인:

### 내용 체크
- [ ] 상황 설명이 충분히 구체적인가?
- [ ] 왜 이 정보가 필요한지 설명되었는가?
- [ ] 질문이 하나의 주제에 집중하는가? (복합 질문 X)
- [ ] 사용자가 이해하기 쉬운 언어인가?
- [ ] 기술 용어에 대한 설명이 있는가?

### 형식 체크 (선택형)
- [ ] 최소 2개 이상의 실질적 옵션이 있는가?
- [ ] 각 옵션의 장단점이 명시되었는가?
- [ ] 구체적인 예시가 포함되었는가?
- [ ] "기타" 옵션이 포함되었는가?

### 형식 체크 (예시 요청형)
- [ ] 원하는 예시 형식이 명확한가?
- [ ] 입력/출력 형식이 제시되었는가?
- [ ] 특수 케이스 예시가 있는가?

### 형식 체크 (개방형)
- [ ] 답변 가이드가 제공되었는가?
- [ ] 고려사항이 구체적인가?
- [ ] 정말로 선택형으로 전환할 수 없는가? (재검토)

### 영향도 체크
- [ ] 이 결정이 영향을 미치는 부분이 명시되었는가?
- [ ] 다른 질문과의 의존성이 표시되었는가?
- [ ] 우선순위 태그가 적절한가?

### 참조 체크 (모듈화 프로젝트)
- [ ] 대상 문서가 명시되었는가?
- [ ] 어느 섹션에 해당하는지 명확한가?
- [ ] Shared 이동 여부가 표시되었는가?

## 나쁜 질문 예시 vs 좋은 질문 예시

### ❌ 나쁜 예 1: 너무 모호함

```markdown
### 질문 1: 데이터를 어떻게 처리할까요?

어떤 방식으로 데이터를 처리하시겠습니까?
```

**문제점**:
- 상황 설명 없음
- 질문이 너무 광범위
- 옵션이나 가이드 없음
- 영향도 미명시

---

### ✅ 좋은 예 1: 구체적이고 명확함

```markdown
### [데이터 처리] 질문 1: 데이터 검증 시점 [High]

**상황**:
"입력 데이터를 검증한다"고 되어 있으나, 검증을 언제 수행할지 불명확합니다.

**질문**:
데이터 검증을 언제 수행해야 할까요?

**옵션**:

A. 파일 로드 즉시 (권장)
   - 장점: 잘못된 데이터 조기 발견, 처리 시간 절약
   - 단점: 초기 로딩 시간 증가
   - 예시: 파일 열기 → 검증 → 실패 시 에러 반환
   
B. 처리 직전
   - 장점: 로딩 빠름, 필요한 데이터만 검증
   - 단점: 중간에 에러 발생 가능, 일부 처리 후 실패
   - 예시: 파일 열기 → 각 레코드 처리 시 검증
   
C. 처리 후 (검증만)
   - 장점: 처리 속도 최대
   - 단점: 잘못된 데이터로 처리 후 발견, 재작업 필요
   - 예시: 전체 처리 → 결과 검증 → 오류 발견 시 재처리

**이 결정이 영향을 미치는 부분**:
- 처리 플로우: 검증 단계 위치
- 에러 처리: 검증 실패 시 복구 방법
- 성능: 전체 처리 시간
- 사용자 경험: 에러 피드백 타이밍
```

---

### ❌ 나쁜 예 2: 복합 질문

```markdown
### 질문 2: 파일 형식과 출력 위치는?

입력 파일 형식은 무엇이고 출력 파일은 어디에 저장하나요?
```

**문제점**:
- 두 개의 독립적인 질문을 하나로 합침
- 각각 다른 우선순위일 수 있음
- 답변하기 어려움

---

### ✅ 좋은 예 2: 분리된 질문

```markdown
### [입력] 질문 2: 입력 파일 형식 [Critical]
...

### [출력] 질문 3: 출력 파일 위치 결정 방식 [High]
...
```

---

### ❌ 나쁜 예 3: 기술 용어 과다

```markdown
### 질문 3: 직렬화 메커니즘

엔티티의 직렬화에 프로토콜 버퍼를 사용할지 JSON을 사용할지, 
아니면 MessagePack를 사용할지 결정해주세요.
```

**문제점**:
- 용어 설명 없음
- 옵션의 차이점 불명확
- 사용자가 기술적 배경 없으면 답변 어려움

---

### ✅ 좋은 예 3: 이해하기 쉬운 설명

```markdown
### [데이터 형식] 질문 3: 데이터 저장 형식 [High]

**상황**:
데이터를 파일에 저장할 때 어떤 형식을 사용할지 결정이 필요합니다.

**질문**:
데이터를 어떤 형식으로 저장할까요?

**옵션**:

A. JSON (사람이 읽을 수 있는 텍스트)
   - 장점: 사람이 직접 읽고 수정 가능, 디버깅 쉬움
   - 단점: 파일 크기 큼, 처리 속도 느림
   - 예시: {"name": "홍길동", "age": 30}
   
B. 바이너리 (최적화된 형식)
   - 장점: 파일 크기 작음, 처리 빠름
   - 단점: 사람이 직접 읽기 어려움, 전용 도구 필요
   - 예시: [바이너리 데이터, 텍스트 편집기로 열면 깨짐]

C. 둘 다 지원
   - 장점: 유연성 최대
   - 단점: 개발 복잡도 증가

**이 결정이 영향을 미치는 부분**:
- 파일 크기: 대용량 데이터 처리 시 중요
- 개발 난이도: 바이너리는 파서 복잡
- 사용자 편의성: 수동 수정 가능 여부
```

## 특수 상황별 질문 예시

### 상황 1: 사용자가 기술적 배경이 없는 경우

**접근 방법**:
- 기술 용어 최소화
- 일상적인 비유 사용
- 시각적 예시 제공

```markdown
### 질문 1: 데이터 저장 방식 [Critical]

**상황**:
프로그램이 처리한 데이터를 저장해야 합니다.

**질문**:
Google Docs와 Microsoft Word 중 어느 방식이 더 편하신가요?

**옵션**:

A. Google Docs 스타일 (온라인 저장)
   - 어디서든 접근 가능
   - 자동 백업
   - 인터넷 필요
   
B. Microsoft Word 스타일 (로컬 파일)
   - 내 컴퓨터에만 저장
   - 오프라인 작업 가능
   - 직접 백업 필요

이 비유로 설명하면, 실제로는:
- A = 클라우드 데이터베이스
- B = 로컬 파일 시스템
```

---

### 상황 2: 사용자 답변이 불명확했던 경우

**접근 방법**:
- 이전 답변 인용
- 구체적인 시나리오 제시
- Yes/No 질문 사용

```markdown
### [후속] 질문 5: 파일 크기 제한 명확화 [High]

**이전 답변 인용**:
질문 2에서 "큰 파일도 처리 가능해야 한다"고 답변하셨습니다.

**질문**:
"큰 파일"의 구체적인 크기는 어느 정도인가요?

**시나리오 기반 질문**:

시나리오 A: 100MB 파일 처리
→ 가능해야 하나요? [예 / 아니오]

시나리오 B: 1GB 파일 처리
→ 가능해야 하나요? [예 / 아니오]

시나리오 C: 10GB 파일 처리
→ 가능해야 하나요? [예 / 아니오]

**또는 구체적인 수치**:
- 최대 파일 크기: ___ MB
- 일반적인 파일 크기: ___ MB
```

---

### 상황 3: 여러 라운드에 걸쳐 진행 중

**접근 방법**:
- 진행 상황 요약
- 남은 이슈 목록
- 우선순위 재확인

```markdown
## 질문 라운드 3

### 진행 상황 요약

**완료된 주제** (라운드 1-2):
- ✅ 파일 형식 (CSV, JSON)
- ✅ 기본 데이터 구조
- ✅ 주요 기능 플로우
- ✅ 에러 처리 기본 정책

**해결된 불명확 영역**: 12개

### 남은 주제 (5개)

**High (2개)**:
1. 성능 최적화 기준
2. 로깅 상세 수준

**Medium (2개)**:
3. UI 레이아웃 세부
4. 설정 파일 위치

**Low (1개)**:
5. 향후 확장 방향

### 이번 라운드 질문 (3개)

우선순위 High 2개를 먼저 해결하고, Medium 1개를 포함합니다.

#### 질문 1: 성능 최적화 기준 [High]
...
```

## 최종 체크리스트

질문지를 제출하기 전 최종 확인:

- [ ] 질문 개수가 5-7개인가?
- [ ] 우선순위 순으로 정렬되었는가?
- [ ] 모든 질문에 우선순위 태그가 있는가?
- [ ] 각 질문이 독립적인가? (복합 질문 없음)
- [ ] 모든 선택형 질문에 2개 이상 옵션이 있는가?
- [ ] 영향도가 모든 질문에 명시되었는가?
- [ ] 기술 용어가 설명되었는가?
- [ ] 문서별로 그룹화되었는가? (모듈화 프로젝트)
- [ ] 질문 간 의존성이 표시되었는가?
- [ ] 사용자가 답변하기 쉬운 형태인가? 또한 사용자가 답변하기 쉽도록 문서 파일로 생성해서 사용자에게 제공했는가?
