# Analyze Scope - 규모 평가 및 문서 구조 결정

## 목적
PRD 초안을 분석하여 프로젝트의 복잡도를 평가하고, 단일 문서로 작성할지 여러 문서로 모듈화할지 결정합니다.

## 사용 시기
- PRD 초안을 처음 받았을 때
- 프로젝트 규모를 판단해야 할 때
- 문서 구조를 결정해야 할 때
- 사용자가 "이 프로젝트는 몇 개 문서로 나눠야 할까?" 같은 질문을 할 때

## 규모 평가 기준

### 소규모 프로젝트 → 단일 문서

**판단 기준**:
- 주요 기능: 1-3개
- 예상 PRD 분량: 5페이지 이하
- 기능 간 의존성: 낮음
- 공통 데이터 구조: 거의 없음

**예시**:
- CSV 파일 형식 변환기
- 단순 API 클라이언트
- 텍스트 처리 유틸리티
- 파일 이름 일괄 변경 도구

**권장 구조**:
```
project-name/
└── PRD.md          # 단일 문서
```

---

### 중규모 프로젝트 → 2-3개 문서

**판단 기준**:
- 주요 기능: 3-4개
- 예상 PRD 분량: 5-15페이지
- 기능 간 의존성: 중간
- 공통 데이터 구조: 일부 존재

**예시**:
- 다국어 번역 도구 (번역 + 검증)
- 파일 관리 유틸리티 (업로드 + 메타데이터 관리)
- 데이터 분석 도구 (수집 + 분석 + 시각화)

**권장 구조**:
```
project-name/
├── PRD-Overview.md    # 전체 개요
├── PRD-Shared.md      # 공통 요소 (선택적)
└── PRD-Features.md    # 모든 기능 (하나의 문서에)
```

또는

```
project-name/
├── PRD-Overview.md       # 전체 개요
├── PRD-Feature-A.md      # 주요 기능 A
└── PRD-Feature-B.md      # 주요 기능 B
```

---

### 대규모 프로젝트 → 모듈화 필수

**판단 기준**:
- 주요 기능: 4개 이상
- 예상 PRD 분량: 15페이지 이상
- 기능 간 의존성: 높음
- 공통 데이터 구조: 다수

**예시**:
- 통합 로컬라이제이션 플랫폼
- 종합 프로젝트 관리 도구
- 전사적 워크플로우 시스템
- 멀티 모듈 개발 도구

**필수 구조**:
```
project-name/
├── PRD-Master.md          # 전체 개요, 아키텍처, 문서 맵
├── PRD-Shared.md          # 공통 데이터 구조, 용어집, 기술 스택
├── PRD-Feature-A.md       # 기능 A 상세 명세
├── PRD-Feature-B.md       # 기능 B 상세 명세
├── PRD-Feature-C.md       # 기능 C 상세 명세
└── PRD-Feature-D.md       # 기능 D 상세 명세
```

## 평가 프로세스

### Step 1: 기능 개수 파악

PRD 초안에서 주요 기능을 식별합니다.

**주요 기능 판단 기준**:
- 독립적으로 사용 가능한가?
- 별도의 UI/입출력이 있는가?
- 10줄 이상의 설명이 필요한가?
- 다른 기능과 명확히 구분되는가?

**예시**:
```
사용자 초안:
"게임 텍스트를 여러 언어로 번역하고, 품질을 검증하고, 
버전을 관리하고, 리포트를 생성하는 도구"

식별된 기능:
1. 다국어 번역
2. 품질 검증
3. 버전 관리
4. 리포트 생성

→ 4개 기능 = 대규모 프로젝트
```

---

### Step 2: 예상 분량 추정

각 기능당 필요한 설명 분량을 추정합니다.

**추정 가이드**:
- 간단한 기능: 1-2페이지
- 중간 복잡도: 3-5페이지
- 복잡한 기능: 5-10페이지

**예시**:
```
기능 1 (번역): 5페이지 (LLM 통합, 프롬프트, 에러 처리 등)
기능 2 (검증): 3페이지 (규칙 엔진, 리포트 등)
기능 3 (버전): 4페이지 (Git 스타일, Diff/Merge 등)
기능 4 (리포트): 2페이지 (템플릿, 내보내기 등)

합계: 14페이지 + 공통 요소 3페이지 = 17페이지
→ 대규모 프로젝트
```

---

### Step 3: 의존성 분석

기능 간 의존 관계를 파악합니다.

**의존성 유형**:
1. **데이터 의존**: 같은 데이터 구조를 공유
2. **순서 의존**: A가 완료되어야 B 실행 가능
3. **기능 의존**: A의 결과를 B가 입력으로 사용

**분석 방법**:
```markdown
## 의존성 매트릭스

|        | 기능 A | 기능 B | 기능 C | 기능 D |
|--------|--------|--------|--------|--------|
| 기능 A |   -    |   →    |        |   →    |
| 기능 B |        |   -    |   →    |        |
| 기능 C |        |        |   -    |   →    |
| 기능 D |        |        |        |   -    |

→: 의존 관계 존재 (A → B: B가 A에 의존)

의존성 점수: 3개 / 12개(3x4) = 25%
→ 중간 수준 의존성
```

**판단 기준**:
- 의존성 < 20%: 낮음 → 독립적 기능들
- 의존성 20-50%: 중간 → 일부 연결
- 의존성 > 50%: 높음 → 밀접하게 연결

---

### Step 4: 공통 요소 식별

여러 기능에서 공유하는 요소를 찾습니다.

**공통 요소 체크리스트**:
- [ ] 데이터 구조 (같은 구조를 2개 이상 기능에서 사용)
- [ ] 용어 (같은 용어가 반복적으로 등장)
- [ ] API/인증 (공통 인증 메커니즘)
- [ ] 파일 형식 (여러 기능에서 같은 형식 처리)
- [ ] 에러 처리 (공통 에러 코드 체계)
- [ ] UI/UX 패턴 (일관된 디자인)

**예시**:
```markdown
## 공통 요소 분석

### 데이터 구조
- TranslationUnit: 기능 A, B, C에서 사용
- Project: 기능 A, C, D에서 사용
- Language: 기능 A, B에서 사용

### 용어
- Source, Target, Segment (모든 기능)
- Quality Score (기능 A, B)

### 기술
- LLM API (기능 A)
- Database (모든 기능)

공통 요소: 6개 이상
→ Shared 문서 필요
```

---

### Step 5: 최종 판단 및 제안

위 4개 단계의 결과를 종합하여 문서 구조를 제안합니다.

**판단 흐름차트**:
```
기능 개수 1-3개?
  ├─ Yes → 단일 문서
  └─ No → 기능 개수 3-4개?
           ├─ Yes → 분량 < 15페이지?
           │         ├─ Yes → 2-3개 문서
           │         └─ No → 모듈화
           └─ No (4개 이상) → 모듈화
```

## 출력 형식

### 소규모 프로젝트 판단 시

```markdown
## 프로젝트 규모 평가

### 분석 결과
- **주요 기능**: [N]개
- **예상 PRD 분량**: [X]페이지
- **기능 간 의존성**: 낮음
- **공통 요소**: 거의 없음

### 판정: 소규모 프로젝트 → 단일 문서

### 권장 문서 구조
```
project-name/
└── PRD.md
```

**PRD.md 포함 내용**:
- 프로젝트 개요 및 목표
- 모든 기능 설명 (한 문서 내 섹션별로)
- 데이터 구조
- 에러 처리
- 테스트 케이스

**장점**:
- 관리 단순
- 전체 내용을 한눈에 파악
- 문서 간 이동 불필요

**다음 단계**: 
이 구조로 진행하시겠습니까? 동의하시면 PRD 정제를 시작하겠습니다.
```

### 중규모 프로젝트 판단 시

```markdown
## 프로젝트 규모 평가

### 분석 결과
- **주요 기능**: [N]개
- **예상 PRD 분량**: [X]페이지
- **기능 간 의존성**: 중간
- **공통 요소**: [Y]개

### 판정: 중규모 프로젝트 → 2-3개 문서

### 권장 문서 구조 (옵션 제공)

#### 옵션 A: 개요 + 통합 기능 문서
```
project-name/
├── PRD-Overview.md    # 전체 개요
└── PRD-Features.md    # 모든 기능
```

**장점**: 관리 단순, 기능 간 비교 용이
**단점**: Features 문서가 길어질 수 있음

#### 옵션 B: 기능별 분리 (권장)
```
project-name/
├── PRD-Overview.md       # 전체 개요
├── PRD-Feature-A.md      # 기능 A
└── PRD-Feature-B.md      # 기능 B
```

**장점**: 기능별 독립적 관리, 병렬 작업 가능
**단점**: 공통 요소 관리 필요

**질문**: 어느 옵션을 선호하시나요? 
또는 다른 구조를 원하시면 말씀해주세요.
```

---

### 대규모 프로젝트 판단 시

```markdown
## 프로젝트 규모 평가

### 분석 결과
- **주요 기능**: [N]개
- **예상 PRD 분량**: [X]페이지 이상
- **기능 간 의존성**: 높음
- **공통 요소**: [Y]개 (다수)

### 판정: 대규모 프로젝트 → 모듈화 필수

### 필수 문서 구조

```
project-name/
├── PRD-Master.md          # 1. 전체 조율 문서
├── PRD-Shared.md          # 2. 공통 요소 저장소
├── PRD-Feature-A.md       # 3. 기능 A
├── PRD-Feature-B.md       # 4. 기능 B
├── PRD-Feature-C.md       # 5. 기능 C
└── PRD-Feature-D.md       # 6. 기능 D
```

### 각 문서의 역할

#### 1. PRD-Master.md (필수)
**역할**: 전체 프로젝트를 조망하는 최상위 문서

**포함 내용**:
- 프로젝트 비전 및 목표
- 전체 시스템 아키텍처
- 기능 목록 및 우선순위
- **문서 맵** (네비게이션)
- 개발 로드맵
- 제외 사항 (Out of Scope)

#### 2. PRD-Shared.md (필수)
**역할**: 모든 기능이 공유하는 요소를 한 곳에 정의

**포함 내용**:
- 공통 데이터 구조 ([Y]개 식별됨)
- 용어집 (Glossary)
- 기술 스택 및 라이브러리
- 공통 API 인증
- 공통 에러 코드
- 공통 파일 형식

**중요**: Feature 문서는 이 문서를 **참조만** 하고 중복 작성 금지

#### 3-6. PRD-Feature-[Name].md (각 기능별)
**역할**: 개별 기능의 상세 명세

**포함 내용**:
- 기능 개요 및 목표
- 상세 요구사항
- 입출력 명세 (Shared 참조)
- 기능별 로직
- 다른 기능과의 의존성
- 기능별 테스트 케이스

### 참조 구조 다이어그램

```
PRD-Master.md (최상위)
     ↓
PRD-Shared.md (공통 저장소)
     ↓
     ├─→ PRD-Feature-A.md (Shared 참조)
     ├─→ PRD-Feature-B.md (Shared 참조)
     ├─→ PRD-Feature-C.md (Shared + A 참조)
     └─→ PRD-Feature-D.md (Shared 참조)
```

### 의존성 관계

식별된 의존성:
- 기능 C → 기능 A (C가 A의 결과 사용)
- 모든 기능 → Shared (공통 데이터 구조 사용)

### 장점
- ✅ 기능별 독립적 개발 가능
- ✅ 중복 제거 (Shared에 집중)
- ✅ 팀원별 문서 분담 가능
- ✅ 관리 공수 최소화
- ✅ 향후 확장 용이

### 주의사항
- 문서 간 동기화 필요 (주 1회 권장)
- 참조 무결성 유지 중요
- Master 문서를 최신 상태로 유지

**질문**: 이 문서 구조에 동의하시나요? 
수정이나 추가할 내용이 있으면 알려주세요.
```

## 특수 상황

### 상황 1: 기능 개수는 적지만 매우 복잡한 경우

**예시**: 기능 2개이지만 각각 10페이지 이상 설명 필요

**판단**:
```markdown
기능 개수: 2개 (소규모 기준)
예상 분량: 25페이지 (대규모 수준)

→ 예외적으로 모듈화 권장

이유: 단일 문서가 너무 길어지면 가독성 저하
```

---

### 상황 2: 기능은 많지만 매우 단순한 경우

**예시**: 기능 5개이지만 각각 1페이지 이하

**판단**:
```markdown
기능 개수: 5개 (대규모 기준)
예상 분량: 8페이지 (소규모 수준)

→ 예외적으로 단일/중규모 문서 가능

이유: 모듈화 오버헤드가 더 클 수 있음
```

---

### 상황 3: 불확실한 경우

**대응**:
```markdown
## 규모 평가 - 추가 정보 필요

현재까지 분석:
- 기능 개수: [N]개
- 예상 분량: 불명확
- 의존성: 불명확

**질문**: 규모를 정확히 판단하기 위해 다음 정보가 필요합니다:

1. 각 기능의 복잡도는 어느 정도인가요?
   - 간단 (1-2페이지 설명 가능)
   - 중간 (3-5페이지)
   - 복잡 (5페이지 이상)

2. 기능 간 데이터 공유가 있나요?
   - 각 기능이 독립적인 데이터 사용
   - 일부 데이터 구조 공유
   - 대부분의 데이터 구조 공유

3. 이 프로젝트는 향후 확장 가능성이 있나요?
   - 현재 범위로 완결
   - 향후 기능 추가 가능성 높음

답변해주시면 정확한 문서 구조를 제안하겠습니다.
```

## 체크리스트

규모 평가 전에 다음을 확인:

- [ ] PRD 초안을 충분히 읽었는가?
- [ ] 모든 기능을 식별했는가?
- [ ] 각 기능의 복잡도를 추정했는가?
- [ ] 기능 간 의존성을 파악했는가?
- [ ] 공통 요소를 식별했는가?
- [ ] 특수 상황에 해당하는지 확인했는가?

규모 평가 후 다음을 제공:

- [ ] 명확한 판정 (소/중/대규모)
- [ ] 구체적인 문서 구조 제안
- [ ] 각 문서의 역할 설명
- [ ] 참조 구조 다이어그램 (대규모인 경우)
- [ ] 장점 및 주의사항
- [ ] 사용자 확인 요청
