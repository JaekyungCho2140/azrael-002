# PRD Refinement Skill

## 목적
초기 PRD 초안을 반복적인 질문-답변 프로세스를 통해 정제하여, LLM이 자의적/임의적으로 해석할 수 없을 만큼 상세하고 명확한 최종 PRD를 작성합니다. 이를 통해 도구 개발 중 방향성이 변경되는 것을 방지합니다.

대규모 통합 도구의 경우, 기능별로 PRD를 모듈화하고 문서 간 참조 시스템을 통해 중복을 제거하여 관리 공수를 최소화합니다.

## 사용 시기
- 새로운 도구나 기능을 개발하기 위한 PRD 작성 시
- 기존 PRD가 모호하거나 불완전하여 보완이 필요할 때
- 여러 기능을 가진 통합 도구의 PRD를 체계적으로 작성할 때
- 사용자가 "PRD를 정제해줘", "PRD를 더 명확하게 만들어줘" 등의 요청을 할 때

## 하위 스킬 구조

이 스킬은 다음의 하위 스킬들로 구성됩니다:

```
prd-refinement/
├── SKILL.md                      # 이 문서 (메인 오케스트레이터)
├── analyze-scope.md              # Phase 0: 규모 평가 및 문서 구조 결정
├── question-generation.md        # Phase 2: 질문지 작성 방법론
├── reference-management.md       # Phase 3-4: 참조 무결성 및 문서 관리
└── templates.md                  # 문서 템플릿 모음
```

각 하위 스킬은 독립적으로 호출 가능하며, 필요에 따라 선택적으로 사용할 수 있습니다.

## 핵심 원칙

### 1. 단계적 정제 프로세스
이 스킬은 다음의 반복적 사이클을 따릅니다:

1. **Phase 0**: PRD 초안 검토 및 규모 평가 → `analyze-scope.md` 참조
2. **Phase 1**: 불명확한 영역 식별 (이 문서)
3. **Phase 2**: 구조화된 질문지 작성 → `question-generation.md` 참조
4. **Phase 3**: 사용자 답변 수집 및 PRD 업데이트 → `reference-management.md` 참조
5. **Phase 4**: 완성도 평가 및 반복 여부 결정 (이 문서)

### 2. 질문의 우선순위
다음 순서로 불명확성을 해소합니다:
- **Critical**: 핵심 기능과 목적에 관한 모호성
- **High**: 데이터 구조, 입출력 형식, 주요 로직
- **Medium**: UI/UX 세부사항, 에러 처리 방식
- **Low**: 최적화, 부가 기능, 향후 확장성

### 3. 모듈화 전략
- 4개 이상의 주요 기능 → 문서 모듈화 필수
- 공통 요소는 Shared 문서에 집중
- Feature 문서는 Shared를 참조만
- 자세한 내용은 `analyze-scope.md` 참조

### 4. 참조 무결성
- 중복 작성 절대 금지
- 참조 링크 형식: `[텍스트](문서명.md#섹션-id)`
- 정기적 무결성 체크
- 자세한 내용은 `reference-management.md` 참조

## Phase 0: 규모 평가 및 문서 구조 결정

**목적**: PRD의 복잡도를 평가하고 적절한 문서 구조를 제안

**실행 방법**: 
```
이 단계의 상세 내용은 analyze-scope.md를 참조하세요.
```

**주요 작업**:
1. 기능 개수 파악
2. 예상 PRD 분량 추정
3. 기능 간 의존성 분석
4. 문서 구조 제안 (단일 vs 모듈화)
5. 사용자 승인 받기

**결과물**: 
- 문서 구조 계획
- 파일 목록 및 각 파일의 역할

## Phase 1: 초기 PRD 분석

PRD 초안 또는 업데이트된 PRD를 다음 관점에서 분석합니다.

### 1.1 단일 문서 프로젝트의 경우

다음 체크리스트로 분석:

- [ ] 도구의 핵심 목적이 명확한가?
- [ ] 해결하려는 문제가 구체적으로 정의되었는가?
- [ ] 각 기능의 입력값이 명확한가?
- [ ] 각 기능의 출력값과 형식이 명확한가?
- [ ] 데이터 구조가 정의되었는가?
- [ ] 동작 로직의 단계가 명확한가?
- [ ] 엣지 케이스 처리 방법이 명시되었는가?
- [ ] 에러 처리 방식이 정의되었는가?

### 1.2 모듈화 프로젝트의 경우

각 문서 유형별로 분석:

#### Master 문서 분석
- [ ] 프로젝트의 핵심 목적이 명확한가?
- [ ] 전체 아키텍처가 정의되었는가?
- [ ] 모든 기능이 식별되고 나열되었는가?
- [ ] 기능 간 우선순위가 명확한가?
- [ ] 문서 맵이 완전한가?
- [ ] 제외 사항(out of scope)이 명시되었는가?

#### Shared 문서 분석
- [ ] 공통 데이터 구조가 정의되었는가?
- [ ] 모든 공통 용어가 용어집에 포함되었는가?
- [ ] 기술 스택이 명확히 선정되었는가?
- [ ] 공통 에러 처리 정책이 있는가?
- [ ] 공통 설정 항목이 정의되었는가?

#### Feature 문서 분석
각 기능 문서에 대해:
- [ ] 기능의 목적이 명확한가?
- [ ] 입력값과 출력값이 명확한가?
- [ ] 기능의 로직이 단계별로 정의되었는가?
- [ ] Shared 문서의 공통 요소를 적절히 참조하는가?
- [ ] 다른 기능과의 의존성이 명시되었는가?
- [ ] 엣지 케이스 처리가 정의되었는가?

#### 참조 무결성 검사
- [ ] 모든 참조 링크가 유효한가?
- [ ] 참조된 섹션이 실제로 존재하는가?
- [ ] 중복된 내용이 여러 문서에 작성되지 않았는가?
- [ ] 공통 요소가 Shared 문서에 집중되어 있는가?

### 1.3 불명확 영역 식별 및 우선순위 부여

분석 결과를 다음 형식으로 정리:

```markdown
## 초기 PRD 분석 결과

### 명확한 부분 ✓
- [잘 정의된 영역 1]
- [잘 정의된 영역 2]
- [잘 정의된 영역 3]

### 불명확한 부분 ⚠️

#### Critical (즉시 해결 필요)
1. [불명확한 영역 1] - [영향도 설명]
2. [불명확한 영역 2] - [영향도 설명]

#### High (우선 해결)
1. [불명확한 영역 3] - [영향도 설명]
2. [불명확한 영역 4] - [영향도 설명]

#### Medium (중요)
1. [불명확한 영역 5] - [영향도 설명]

#### Low (선택적)
1. [불명확한 영역 6] - [영향도 설명]
```

## Phase 2: 질문지 작성

**목적**: 식별된 불명확성을 해소하기 위한 구조화된 질문 생성

**실행 방법**: 
```
이 단계의 상세 내용은 question-generation.md를 참조하세요.
```

**주요 원칙**:
- 한 번에 최대 5-7개 질문
- 우선순위가 높은 것부터
- 구체적인 옵션 제공
- 영향 범위 명시
- 참조 구조 제안 (모듈화 프로젝트)

## Phase 3: 답변 통합 및 PRD 업데이트

**목적**: 사용자 답변을 바탕으로 PRD 업데이트 및 참조 무결성 보장

**실행 방법**: 
```
이 단계의 상세 내용은 reference-management.md를 참조하세요.
```

**주요 작업**:
1. 답변 요약 및 확인
2. 문서별 업데이트 계획 수립
3. PRD 업데이트 실행
4. 참조 무결성 보장
5. 변경 이력 기록

## Phase 4: 완성도 평가

업데이트된 PRD에 대해 다음을 평가합니다:

### 4.1 완성도 체크리스트

#### 단일 문서 프로젝트

```markdown
## PRD 완성도 체크리스트

### 명확성 (Clarity)
- [ ] 모든 요구사항이 구체적이고 측정 가능한가?
- [ ] "적절히", "필요시", "가능하면" 등의 모호한 표현이 없는가?
- [ ] 예시나 구체적인 값이 포함되어 있는가?

### 완전성 (Completeness)
- [ ] 모든 기능의 입출력이 정의되었는가?
- [ ] 엣지 케이스가 식별되고 처리 방법이 명시되었는가?
- [ ] 에러 처리 방식이 구체적으로 정의되었는가?

### 일관성 (Consistency)
- [ ] 용어 사용이 일관적인가?
- [ ] 다른 요구사항과 충돌하는 부분이 없는가?
- [ ] 데이터 형식이 전체적으로 일관적인가?

### 실행 가능성 (Actionability)
- [ ] 개발자가 이 PRD만으로 구현을 시작할 수 있는가?
- [ ] 추가 질문이나 가정 없이 개발 가능한가?
- [ ] 테스트 케이스 작성이 가능한 수준인가?

### 비모호성 (Unambiguity)
- [ ] LLM이 다르게 해석할 수 있는 부분이 없는가?
- [ ] 여러 구현 방법 중 선택이 필요한 부분이 명시되었는가?
- [ ] 모든 조건과 기준이 구체적인가?
```

#### 모듈화 프로젝트 (추가 체크리스트)

```markdown
### 문서 구조
- [ ] Master 문서에 모든 기능이 나열되었는가?
- [ ] 각 기능이 적절한 문서에 배치되었는가?
- [ ] 문서 맵이 최신 상태인가?

### 참조 무결성
- [ ] 중복 내용이 제거되었는가?
- [ ] 공통 요소가 Shared 문서에 집중되었는가?
- [ ] 모든 참조 링크가 유효한가?
- [ ] 참조가 필요한 곳에 적절히 사용되었는가?

### 유지보수성
- [ ] 문서 구조가 향후 확장을 고려하는가?
- [ ] 변경 이력이 각 문서에 기록되는가?
- [ ] 관리 공수가 최소화되도록 설계되었는가?
```

### 4.2 반복 여부 결정

```markdown
## 다음 단계 결정

**현재 완성도**: [N]%

**남은 불명확 영역**:

### Critical
- [없음 / 영역 나열]

### High
- [없음 / 영역 나열]

### Medium
- [없음 / 영역 나열]

### Low
- [없음 / 영역 나열]

**권장 조치**:
- [ ] 추가 질문 라운드 필요 (Critical/High 영역이 남아있음)
- [ ] 마이너 수정 후 완료 가능 (Medium/Low 영역만 남음)
- [ ] PRD 작성 완료 (모든 영역이 충분히 명확함)

**다음 라운드 우선순위** (추가 라운드가 필요한 경우):
1. [영역] - [우선순위] - [이유]
2. [영역] - [우선순위] - [이유]
```

```markdown
3. [개선사항 3]

---
**다음 단계**: 이 PRD를 바탕으로 개발을 시작할 수 있습니다.
```

## 베스트 프랙티스

### DO ✓
- 한 번에 하나의 측면에 집중하여 질문
- 구체적인 예시와 옵션 제공
- 각 결정이 미치는 영향 명시
- 사용자의 답변을 요약하여 확인
- 우선순위가 높은 불명확성부터 해소
- 이전 답변과의 일관성 확인
- 하위 스킬을 적극 활용 (모듈화의 이점)

### DON'T ✗
- 모호하거나 추상적인 질문
- 이미 명확한 부분에 대한 중복 질문
- 사용자가 답변하지 않은 부분에 대한 임의 가정
- 이전 답변을 무시한 질문
- 하위 스킬의 내용을 메인 스킬에 중복 작성

## 종료 조건

다음 조건을 모두 만족하면 PRD 정제를 완료합니다:

### 단일 문서 프로젝트
1. ✓ 모든 Critical과 High 우선순위 불명확성 해소
2. ✓ 주요 기능의 입출력이 구체적으로 정의됨
3. ✓ 엣지 케이스와 에러 처리 방식 명시됨
4. ✓ 개발자가 추가 가정 없이 구현 가능한 수준
5. ✓ 사용자가 추가 질문이 없다고 확인

### 모듈화 프로젝트 (추가 조건)
6. ✓ Master 문서가 전체를 조율하는 역할을 함
7. ✓ Shared 문서에 모든 공통 요소가 집중됨
8. ✓ Feature 문서 간 중복이 제거됨
9. ✓ 참조 무결성이 보장됨
10. ✓ 문서 구조가 향후 확장을 고려함

## 주의사항

1. **과도한 정제 방지**: 완벽을 추구하다 실행이 지연되지 않도록 합니다. 80-90% 명확성에 도달하면 개발 시작을 고려합니다.

2. **컨텍스트 유지**: 여러 라운드에 걸쳐 진행되므로, 이전 답변과 결정사항을 계속 참조합니다.

3. **유연성**: 사용자가 예상치 못한 답변을 하거나 방향을 바꾸려 할 때 유연하게 대응합니다.

4. **모듈화의 장점 활용**: 복잡한 작업은 하위 스킬로 위임하여 메인 스킬을 단순하게 유지합니다.

## 하위 스킬 호출 가이드

### 언제 어떤 스킬을 호출하는가

**analyze-scope.md**:
- PRD 초안을 처음 받았을 때
- 프로젝트 규모를 판단해야 할 때
- 문서 구조를 결정해야 할 때

**question-generation.md**:
- 불명확 영역에 대한 질문을 작성할 때
- 질문 형식이나 우선순위 결정에 도움이 필요할 때
- 참조 관련 질문을 작성할 때 (모듈화 프로젝트)

**reference-management.md**:
- PRD를 업데이트할 때
- 참조 무결성을 체크해야 할 때
- 중복 내용을 발견했을 때
- 변경 이력을 기록해야 할 때

**templates.md**:
- 새 문서를 생성해야 할 때
- 문서 형식이 필요할 때
- 체크리스트나 표준 구조가 필요할 때

## 도구와의 통합

이 스킬은 다음과 같은 Claude Code 워크플로우와 함께 사용됩니다:

```
1. 사용자: 간단한 PRD 초안 제공
2. Claude: Phase 0 (analyze-scope.md) - 규모 평가
3. Claude: Phase 1 - 불명확 영역 식별
4. Claude: Phase 2 (question-generation.md) - 질문 생성
5. 사용자: 질문에 답변
6. Claude: Phase 3 (reference-management.md) - PRD 업데이트
7. Claude: Phase 4 - 완성도 평가
   ↓
   완료? No → 3단계로 돌아가기 (다음 라운드)
   완료? Yes → 최종 PRD 제공
```
